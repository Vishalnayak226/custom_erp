# Hardening Roadmap: Toward a Lightweight, Solid, Foolproof ERP

This document translates a source-verified code audit (originally `docs/UNDERSTANDING_TEMP.md`, now folded into the permanent docs) into a concrete, phased plan. The goal isn't to add new modules — it's to make what's already built lightweight, smooth, secure, and hard to break.

## What "best ERP out there" means concretely, for this codebase

| Quality | What it means here |
| :--- | :--- |
| **Lightweight** | Keep the Go single-binary + PostgreSQL stack, stdlib `net/http` (no framework), and the current minimal dependency set (`lib/pq`, `golang.org/x/crypto` — both permissively licensed, keep it that way). Add pagination to the one unbounded list endpoint rather than reaching for heavier infrastructure. |
| **Smooth** | Finish the custom-dialog migration — `showCustomAlert()`/`showCustomConfirm()` exist and are used app-wide, but several `prompt()` calls (DocType Builder, Prefix Config edit) are still raw browser prompts. A `showCustomPrompt()` equivalent would make the UX consistent everywhere. |
| **Not copyright** | Dependencies are already permissively licensed (MIT/BSD) — no action needed. The DocType/metadata-driven pattern is *inspired by* Frappe/ERPNext/Odoo/Nocobase (architectural ideas, not copyrightable) — no code or branding was copied from them, which is the right posture to maintain going forward. Separately, as a business (not code) consideration: "Custom ERP" is a generic placeholder name — worth a distinct name and trademark check before any public release, but that's a product decision, not something to change unilaterally in code. |
| **Easy** | Already strong via the DocType Builder + dynamic label engine, which let new fields/doctypes appear in the UI without recompiling. Complements: finish the `prompt()` → custom modal migration (see "Smooth"), and eventually build the spec'd-but-missing Log Hub UI so admins get self-service visibility into errors/integration failures without needing a developer.
| **Lifelong solid / never breakable / foolproof / secure** | The substance of this roadmap — four phases below, security first. |

## Phase 1 — Critical Security

The highest-priority phase. Each item below is a specific, verified issue, not a hypothetical:

1. ✅ **DONE (2026-07-12). Removed the silent "no auth header → admin" fallback.** `apiMiddleware` in `main.go` now rejects any request with no/invalid `Authorization` header (except `POST /api/v1/login` itself, which needs to be reachable pre-auth). This turned out to require more than a one-line backend change: the frontend never had a login screen and relied entirely on the fallback to function, and the seeded users' password hash had no known plaintext anywhere. Also fixed as part of this item:
   - Built a real login screen (`public/index.html`/`app.js`/`styles.css`) — username/password form, token stored in `localStorage`, global 401-handling now logs out and returns to the login screen instead of just alerting, added a logout button in the sidebar.
   - Reset all 4 seed users (`admin`, `cashier1`, `manager1`, `system`) to fresh, unique bcrypt hashes (`db/migration.sql`) — the old hash's plaintext was never recorded anywhere and is now unrecoverable/irrelevant. New dev-only plaintext values are in `DEV_CREDENTIALS.local.txt` (gitignored, project root) — rotate before any non-dev use.
   - Verified end-to-end (unauthenticated → 401, login endpoint reachable pre-auth, wrong password rejected, correct password issues a token, issued token accepted by a protected endpoint) — all passing.
2. ✅ **DONE (2026-07-12). Fixed the SQL injection in the generic document list filter.** `handleGenericDoc`'s dynamic GET-filter loop in `main.go` spliced the *query-parameter key* (not just the value) directly into `data->>'%s'` — unescaped. Fixed by allowlisting the key against a strict identifier pattern (`^[a-zA-Z_][a-zA-Z0-9_]{0,63}$`, package-level `safeFilterKeyRe` in `main.go`) before it ever touches the query string — any key with quotes, parens, semicolons, or other SQL-meaningful characters is rejected with `400 Bad Request` rather than reaching the query. Chose a strict pattern allowlist over checking against `doctype_fields` specifically because field names themselves aren't format-validated at creation time (`SaveFieldDefinition`), so allowlisting against that table wouldn't fully close the hole on its own — the pattern check does, independent of what's in the metadata tables. Verified: two injection-style keys rejected with 400, a legitimate filter (`?status=Active`) still returns 200, reserved keys (`q`, `tenant_id`) still pass through untouched.
3. ✅ **DONE (2026-07-12). Moved the hardcoded auth secret out of source.** `engines/auth.go`'s `jwtSecret` is no longer a literal in code. Deviated from the original "fail fast if unset" wording here on purpose: an explicit `JWT_SECRET` env var still wins if set (production path), but if unset, the server now auto-generates a random secret once and persists it under the OS per-user config dir (`%APPDATA%\custom_erp\jwt_secret.local` on Windows) — never in the repo, never in the project working directory. A true fail-fast would mean re-setting an env var by hand after every restart during iterative dev, invalidating every session each time; this gets the same "no hardcoded secret" outcome without that friction, and it's still overridable for real deployments. Verified: login/token-issuance/protected-endpoint round-trip works against the new persisted secret, and a malformed/stale token still cleanly 401s rather than 500ing.
4. ✅ **DONE (2026-07-12). Added token expiry.** `SignToken` now embeds an `exp` claim (unix timestamp); `ParseToken` rejects tokens with no `exp`, a malformed `exp`, or an `exp` in the past. Default lifetime is 24h (long enough for a shift/session with no refresh-token mechanism in place, short enough that a leaked token doesn't stay valid indefinitely) — overridable via `JWT_EXPIRY_HOURS` for real deployments, same override pattern as `JWT_SECRET`. Verified with a direct unit test (valid token parses, an artificially backdated `exp` is rejected with "token expired", a token missing `exp` entirely is rejected) plus a live HTTP round-trip confirming issued tokens actually carry the claim and are still accepted while fresh.
5. ✅ **DONE (2026-07-12). Replaced Origin-reflecting CORS with an explicit allowlist.** `apiMiddleware` no longer echoes back whatever `Origin` a requester sends. A package-level allowlist (`corsAllowedOrigins` in `main.go`) defaults to the local dev origins (`http://localhost:8080`, `http://127.0.0.1:8080`) and is extended via `CORS_ALLOWED_ORIGINS` (comma-separated) for real deployments — same override pattern as `JWT_SECRET`/`JWT_EXPIRY_HOURS`. An `Origin` not in the allowlist gets none of the `Access-Control-Allow-*` headers at all, so the browser blocks the cross-origin response. Verified live: an allowed origin gets the full CORS header set back, a disallowed origin (`http://evil.com`) gets none, requests with no `Origin` header are unaffected, and the env-var override correctly adds origins (with whitespace trimmed) without disabling the built-in defaults.
6. ✅ **DONE (2026-07-12). Stopped propagating the shared placeholder password hash to new tenants.** `ProvisionTenantSchema` (`engines/saas.go`) no longer clones `tenant_default`'s `users` table data (it still clones the table *structure*, just not the rows). Instead it creates exactly one `admin` user per new tenant with a freshly generated, high-entropy random password, hashed with bcrypt and returned once in the API response (`handleProvisionTenant`) for the caller to capture — same "shown once, never persisted in plaintext, can't be retrieved again" pattern used for the dev seed credentials. `engines_test.go`'s `SaaSProvisioningAndFeatureFlags` subtest now asserts the new tenant's admin hash differs from `tenant_default`'s and that exactly one user gets seeded (not the full `cashier1`/`manager1`/`system` roster).
7. ✅ **DONE (2026-07-12). Added an explicit role check to admin-only routes.** Both `handleProvisionTenant` and `handleSetFeatureFlag` now reject with `403 Forbidden` unless `Resolved-Role == "HR/Admin"`. Fixed together with item 6 since they're the same functional area and shipping one without the other would've left a real gap — any authenticated non-admin (e.g. a Cashier) could otherwise still provision new tenant schemas. Verified live: a Cashier-role token gets 403 from both endpoints, an HR/Admin token succeeds, the returned password actually matches what's stored (bcrypt-verified directly, not re-tested through `/login` to avoid colliding with its own 5/min rate limit), and the new tenant's admin hash differs from `tenant_default`'s.

## Phase 2 — Correctness

1. ✅ **DONE (2026-07-12). Added a stock floor-check to direct checkout.** `PostInventoryLedger` (`engines/inventory.go`) now row-locks (`FOR UPDATE`) and validates available stock before applying any negative delta (checkout/decrement) — if the decrement would push `available` below zero, the whole transaction is rolled back and a clear "insufficient stock" error is returned, rather than silently clamping to zero or going negative. Positive deltas (GRN receiving) are untouched — the check only triggers when `qty < 0`. Chose reject-and-roll-back over clamp-at-zero (the pattern already used elsewhere, e.g. fulfillment dispatch) because clamping silently completes an oversold transaction rather than actually preventing it — the roadmap goal was "can't oversell," not just "can't go negative in the DB." Verified live: a checkout for more than available stock is rejected with stock left untouched, a checkout for exactly the available amount succeeds and leaves stock at exactly 0 (never negative), and a further checkout against zero stock is also correctly rejected. Confirmed neither existing caller of `PostInventoryLedger` (the scale-test simulator, an existing unit test) is affected — both only ever pass positive quantities. Side finding, not fixed: the scale-test's "simulated checkout" branch actually passes a positive `qty`, so despite its comment ("decredits stock"), it's adding stock, not decrementing it — a pre-existing test-fidelity gap, not a production bug.
2. ✅ **DONE (2026-07-12). Fixed the checkout-vs-forecast status mismatch.** Audited every place that sets or reads `POSCart` status before deciding which side to fix: `handleCheckout` writes `'Paid'`, and `ProcessMarketplaceSettlement` (`engines/marketplace.go`) transitions a cart to `'Settled'` after reconciliation — `'completed'` was never a real status this system produces anywhere. Fixed the read side: `CalculateSalesVelocity` (`engines/optimization.go`) now matches `status IN ('Paid', 'Settled')` rather than `'completed'` — both represent a completed sale for velocity purposes, since a settled cart was paid first. Changing the *writer* (`handleCheckout`) would have been riskier and touched more surface area for no benefit. Updated `engines_test.go`'s fixture to insert `'Paid'` (matching real checkout behavior) instead of the fictional `'completed'` it was using to make the old, broken query happy. Verified with the actual bug scenario end-to-end: real `POST /api/v1/checkout` (10 units) followed by real `POST /api/v1/optimization/forecast` now returns `forecasted_demand: 10.0` (10 units ÷ 30-day lookback × 30 forecast days) — previously this exact flow always returned 0 regardless of real sales.
3. ✅ **DONE (2026-07-12). Made the outbox worker multi-tenant.** `StartOutboxWorker` (`engines/outbox.go`) no longer hardcodes `schema := "tenant_default"`. It now queries `SELECT DISTINCT schema_name FROM public.tenants` fresh on every tick and processes each one's outbox — so newly provisioned tenants (via the Phase 1.6/1.7 fix) are picked up automatically with no server restart needed. `tenant_default` itself needs no special-casing since it's already registered there (seeded for `tenant_id = 'default'`). Verified live: provisioned a second tenant via the real API, inserted a `Pending` outbox event directly into its schema, waited for the worker's 5s tick, and confirmed it was picked up and dispatched (previously this would have sat forever, invisible to the worker) — while `tenant_default`'s own outbox processing kept working correctly too.
4. ✅ **DONE (2026-07-12). Added pagination to the generic document list endpoint.** `GET /api/v1/doc/{doctype}` now accepts optional `limit`/`offset` query params (`main.go`) and adds `ORDER BY id LIMIT $n OFFSET $m` to the query. Critically, even when the caller passes neither, a default cap (`defaultListLimit = 500`, hard ceiling `maxListLimit = 1000`) still applies — the endpoint can no longer return a truly unbounded result set either way. Kept the response as a plain JSON array (no envelope change) rather than adding a `{data, total, page}` wrapper, because `public/app.js` already fetches the full array and paginates it client-side (`currentTablePage`/`itemsPerPage`) — changing the shape would have broken the frontend for no benefit; the existing client-side pagination now just operates over a capped-but-still-plenty-large set instead of a literally unbounded one. Documented but deliberately not fixed: when a search term (`q`) is active, the limit/offset bound the SQL-level candidate set fetched *before* the existing in-memory search filter runs, so a search could in principle miss a match sitting past the current page window — moving search into SQL would remove that edge case but is a bigger change than this item calls for. Verified live with 7 isolated seeded records: `limit=3&offset=0`/`3`/`6` return exactly 3/3/1 with zero ID overlap across pages, `limit`/`offset` are correctly excluded from the dynamic JSONB-filter logic (not misread as document fields), and an absurd `limit` value doesn't error out.
5. ✅ **DONE (2026-07-12). Fixed the wrong date-format layout.** `engines/marketplace.go:108` now formats `payout_date` with `time.Now().Format("2006-01-02")` (was `"2006-02-01"`, which actually produces `YYYY-DD-MM`, silently swapping day and month). `go vet ./engines/...` is clean now — this was the warning it had been flagging all through Phase 1.
6. ✅ **DONE (2026-07-12). Fixed `SwitchIndustryProfile` silently deleting field definitions the new profile doesn't redeclare.** It used to run `DELETE FROM doctype_fields WHERE doctype_name = $1` then insert *only* the fields the incoming profile JSON lists — none of the four shipped profiles (jewelry/food_bev/auto/clothing) redeclare base fields like `status`, so every single industry switch was silently deleting it. Changed to upsert (`ON CONFLICT (doctype_name, fieldname) DO UPDATE`) instead of delete-then-insert — chose merge over "warn but still delete" because every shipped profile is authored as a *partial* industry-specific overlay, not a complete field list, so deletion was never really the intended semantics to begin with. Trade-off, accepted deliberately: fields added by one industry (e.g. Jewelry's `designer`) will linger if you later switch to a different industry that doesn't mention it either — a field silently *staying registered* is a far smaller problem than one silently *disappearing and breaking validation*, and per `framework_architecture.md` §7.1 industry switching is meant to be a rare, early-lifecycle action anyway (locked once real transactions exist). Also restored the live dev DB's `Brand.status` field, which a past switch had already dropped before this fix landed — **this is what was causing the `DocTypeValidationAndAuth` test failure that persisted through every step of Phase 1**; the full suite is fully green for the first time this session. Verified live end-to-end on a fully disposable, isolated tenant (not `tenant_default`): `Brand` starts with `code/name/status`, switching to `JEWELRY` leaves `status` intact *and* adds `designer`, and a bad status value is still correctly rejected by the real `POST /api/v1/doc/Brand` endpoint afterward.

## Phase 3 — Test & CI discipline

So that bugs like the Phase 2.2 status mismatch stop slipping through a green test suite:

1. ✅ **DONE (2026-07-12). Added a true integration test that drives the real HTTP handlers end-to-end.** New `main_test.go` (project root, package `main` — has to live there since `engines_test.go` can't import `main`, and only `main` can call unexported handlers like `handleCheckout` directly). `TestCheckoutToForecastIntegration` calls `apiMiddleware(handleLogin)` → `apiMiddleware(handleCheckout)` → `apiMiddleware(handleDemandForecast)` via `httptest.NewRecorder`/`httptest.NewRequest` (in-process, no real socket/port needed), asserts the checkout actually persists `status = 'Paid'`, and asserts the forecast reflects it (20 units sold → `forecasted_demand: 20.0`). Proved this test has real teeth, not just incidental passing: temporarily reverted `CalculateSalesVelocity`'s query back to the exact original bug (`status = 'completed'`), re-ran just this test, confirmed it fails with `expected forecasted_demand ... got 0` — then restored the real fix and confirmed green again. This is now a permanent regression test; the old `AdvancedOptimizationAndForecasting` unit test (raw-SQL fixture, bypasses `handleCheckout`) still exists alongside it for the pure-math coverage, but can no longer be the *only* thing standing between a status-value typo and a silent no-op in production.
2. ✅ **DONE (2026-07-12). Added a minimal CI workflow.** New `.github/workflows/ci.yml` — GitHub Actions (the repo already has a GitHub remote, so this is genuinely zero new infrastructure), runs on every push/PR to `main`: spins up a `postgres:16` service container mapped to port 5435 (matching both test files' hardcoded connection string), applies `db/migration.sql` + `db/migrations_phase3.sql`, then `go build ./...` → `go vet ./...` → `go test ./... -v`. Didn't just write it and assume it'll work: spun up a second, fully isolated local Postgres instance (separate data directory, port 5436, torn down afterward) to simulate exactly what CI does — fresh database, migrations applied from scratch, no accumulated state from this session's live dev DB — and confirmed build/vet/test all pass cleanly against it before writing the workflow file, so the first real push has already effectively been dry-run locally. The live dev database on port 5435 was never touched by this verification.

## Phase 4 — Release hygiene

Closes the gap between what `micro_checklist.md` already claims and what the build actually does:

1. ✅ **DONE (2026-07-12). Stripped release binaries.** Added a `release` action to `manage.ps1` (`go build -ldflags="-s -w" -o erp-server.exe`) rather than just documenting the flag somewhere easy to forget — it stops the server first if running (Windows locks a running `.exe`, the build would otherwise fail) and deliberately does not auto-restart it, matching how every other change this session gets applied. Verified with real, measured numbers: **9,446 KB → 6,602 KB (~30% smaller)**, then confirmed the stripped binary is functionally identical (starts cleanly, unauthenticated requests still correctly 401). Dev builds (`go build -o erp-server.exe`, via `manage.ps1 start`) intentionally stay unstripped — debug symbols are worth keeping during active development for readable panic stack traces.
2. ✅ **DONE (2026-07-12). Stopped shipping JS sourcemaps.** Dropped `--sourcemap` from the `esbuild` invocation in `package.json`'s `build` script. Went further than the minimum fix: `public/dist/` (the build output) was actually already **committed to git** — including the `.map` file — with no `.gitignore` entry at all, so simply changing the script going forward wouldn't have removed what was already tracked. Ran `git rm --cached` on `public/dist/` and added it to `.gitignore` (regeneratable build output belongs there, same category as `node_modules/`/`*.exe`, already-established policy in this repo). Regenerated `dist/` with the fixed script and confirmed directly: zero `.map` files anywhere in the output, and both minified bundles are still syntactically valid (`node --check`). Side note for context, not a live risk today: `public/index.html` loads raw `app.js`/`db.js` directly, not the `dist/` bundle at all — so this was a build-pipeline correctness fix for whenever the minified bundle *does* get wired up for a real release, not something actively exposed right now.

## Sequencing

Each phase is independent and can be implemented and reviewed one at a time. Per the project's working agreement, code changes happen only when explicitly requested, and the server is restarted manually by the user after each applied change — this roadmap is the backlog to pull from, not something to execute automatically.
